# logref-wasm

The **wasm-bindgen binding for logref's real Rust `scan` surface**, generated by
fluessig's `wasm` backend. It exposes `logref-core`'s `lower_format` /
`render_sample` free functions and its stateful `Scanner` (as a handle class with
`scanLine` / `scanBatch` / `scanBatchPacked` / `report`) to JS/TS in the browser —
the *actual* Rust scanner, not a hand-ported mirror.

This started as a proof-of-concept for fluessig's `wasm` backend; it is now the
**production scanner of the site's Scan page** (`site/src/scan.ts`). The
hand-ported TS mirrors it was originally benchmarked against
(`site/src/lower.ts`, `site/src/scanner.ts`) are retired — the Rust crate is the
single source of truth, byte-for-byte.

## How it's built

The scan surface is authored once as a fluessig-derive schema in `crates/logref-schema`;
`scripts/gen.sh` emits the catalog/api JSON and runs `fluessig-gen --wasm` to regenerate
`src/binding.rs` (drift-free). `src/core_impl.rs` is the hand-written seam over
`logref-core`; `src/packed.rs` is the hand-written packed batch-scan surface (one flat
`Int32Array` result — see below); `src/lib.rs` wires them together. The crate is a
`cdylib` built for `wasm32`:

- **for the site**: `scripts/build-wasm-web.sh` (also `bun run build:wasm` from
  `site/`) → web-target glue + `.wasm` into `site/src/wasm/` (simd128 on);
- **for the bench demo**: `bench/build.sh` (`SIMD=1` opts in).

It is kept off the host cargo build (excluded from the root workspace
`default-members`) so host CI's rust job never compiles wasm-bindgen; the site CI
job builds the wasm32 target with a pinned wasm-bindgen CLI.

## Parity — WASM matches Rust exactly

The WASM `Scanner` output matches the native Rust scanner **byte-for-byte** on the
ground-truth fixtures and on a generated 4k-pattern corpus; the specialized matcher
is proven byte-identical to the retained `RegexSet` oracle in-browser (the bench
cross-checks every line) and natively (`cargo test`). See `scan.parity.test.ts`,
`smoke.test.ts` (build the nodejs-target `pkg/` first:
`wasm-bindgen --target nodejs --out-dir crates/logref-wasm/pkg target/wasm32-unknown-unknown/release/logref_wasm.wasm`),
and `site/src/scanwasm.test.ts` for the site-level layer.

## Benchmark — specialized-matcher WASM wins ~10× (headless Chromium, N=20k lines)

At production catalog scale (~4k patterns), after `logref-core` gained the specialized
printf matcher (anchored Aho-Corasick candidates + regex-free byte-loop verification —
see `scan.rs`):

| path                                             | scan throughput  | notes                          |
| ------------------------------------------------ | ---------------- | ------------------------------ |
| specialized WASM (`scanBatchPacked` + JS decode) | ~1.1M lines/sec  | simd128 build; ~1.03M without  |
| specialized WASM (`scanBatch`, nested serde)     | ~680k lines/sec  | marshalling-bound              |
| previous trigram+`regex` WASM (packed)           | ~95k lines/sec   | the path it replaced           |
| hand-tuned TS trigram scanner (retired)          | ~110k lines/sec  | the old site runtime           |

Scanner **build** at 4k patterns is ~50–85 ms in-browser — the oracle-side
`Regex`/`RegexSet` compilation that used to dominate (~1.3 s) is now lazy in
`logref-core` (`OnceLock`, built only when an oracle/trigram cross-check path is
called), so the production path never pays it. Run it yourself: `bench/build.sh`
(`SIMD=1` for the simd128 build) then `bun bench/drive.ts`.

The win is **algorithmic + marshalling**: the specialized matcher verifies only
anchor-confirmed candidates with direct byte loops, and the packed `Int32Array`
result crosses the boundary as one bulk copy (capture *spans*, sliced JS-side by
the caller who already holds the lines) instead of ~75k serde-built JS objects.
