# logref-wasm

A proof-of-concept **wasm-bindgen binding for logref's real Rust `scan` lowering**,
generated by fluessig's new `wasm` backend. It exposes `logref-core`'s `lower_format`
free function and its stateful `Scanner` (as a handle class with `scanLine` / `scanBatch`)
to JS/TS in the browser â€” the *actual* Rust scanner, not a hand-ported mirror.

The point was to prove fluessig's `wasm` backend end-to-end on a real surface, and to
measure whether running logref's Rust scanner in the browser via WASM beats the
hand-tuned TS scanner the site ships today.

## How it's built

The scan surface is authored once as a fluessig-derive schema in `crates/logref-schema`;
`scripts/gen.sh` emits the catalog/api JSON and runs `fluessig-gen --wasm` to regenerate
`src/binding.rs` (drift-free). `src/core_impl.rs` is the hand-written seam over
`logref-core`; `src/lib.rs` wires them together. The crate is a `cdylib` and is built for
`wasm32` via `bench/build.sh` â€” it is kept off the host cargo build (excluded from the
root workspace `default-members`) so host CI never compiles wasm-bindgen.

## Parity â€” WASM matches Rust exactly

- The WASM `Scanner` output matches the native Rust scanner **byte-for-byte** on the
  ground-truth fixtures and on a generated 4k-pattern corpus.
- `lower_format` matches the hand-ported `site/src/lower.ts` `lowerFormat`
  **byte-for-byte**.

See `scan.parity.test.ts` and `smoke.test.ts`.

## Benchmark â€” TS wins (headless Chromium, N=20k lines)

At production catalog scale (~4k patterns):

| path                              | scan throughput   | notes                         |
| --------------------------------- | ----------------- | ----------------------------- |
| batched WASM (`scanBatch`)        | ~59k lines/sec    | ~30Ă— slower to build the index |
| hand-tuned TS trigram scanner     | ~121k lines/sec   | current site runtime          |

**TS wins ~2Ă— on scan and ~30Ă— on build** at production scale. WASM only edges ahead on
a tiny 48-pattern sample.

The gap is **algorithmic + marshalling, not the boundary crossing**: the Rust path runs a
`RegexSet` over *all* patterns, while the TS path uses a trigram prefilter that skips the
vast majority of patterns per line; on top of that, serde has to marshal a deep nested
result across the boundary. Because the cost isn't the JSâ†”WASM call overhead, batching the
op (`scanBatch`) does **not** close the gap. See `bench/README.md` for the harness.

## Recommendation

**Do not swap the runtime `scanner.ts` to WASM** â€” it would regress scan throughput ~2Ă—.

The real fit for WASM / native Rust here is **build-time lowering**: use the Rust
`lower_format` to generate `patterns.json` at build time. That retires the drift risk of
the hand-maintained `site/src/lower.ts` (in particular its hand-rolled `regex::escape`
`META` regex, which must stay in lockstep with Rust's) while keeping the fast TS trigram
scanner for the per-line hot path.
